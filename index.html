The Problem

Why We're Doing This

The Current Reality

Every two weeks, our team manually builds 2 bi-weekly slide decks (Scan and Reader) - each 49-168 slides. This involves:

  • ● Manually pulling data from App Store Connect, Google Play Console, AppTweak
  • ● Manually building FTD bar charts and YTD comparison tables
  • ● Manually inserting keyword rankings, experiment results, and event data
  • ● Manually taking app store screenshots and placing them on slides
  • ● Writing strategic commentary, learnings, and next steps
  • ● Estimated 6-10+ hours per cycle across the team

See the actual decks linked in the reports overview above.

The Vision

A system where AI generates 70-80% of the deck automatically, and consultants focus only on strategic commentary and review:

  • ● Data pulled automatically from BigQuery (console + AppTweak data)
  • ● Charts and tables generated programmatically matching the existing format
  • ● AI drafts initial narrative, analysis, and commentary
  • ● Consultant reviews, edits, and approves (human-in-the-loop)
  • ● Final PPTX generated and ready to send to Adobe
  • ● Estimated 30-60 minutes per cycle (review only)

That's a 90% reduction in manual effort.

Human-in-the-Loop, Not Human-Out-of-the-Loop

Nothing in this system runs without human review. AI generates the first draft - data, charts, and narrative - but a consultant always reviews, edits, and approves before anything goes to Adobe. The goal is to eliminate tedious data gathering and formatting, not strategic thinking.

Creative Workarounds

Solving the Hard Parts

App Store Screenshots

Current: manually taken and inserted. Appears in experiment results and market-specific updates.

Solution: Playwright headless browser to capture app store listings programmatically. Schedule nightly for key markets. Fallback: AppTweak may provide screenshot endpoints.

Experiment Creative Variants

Current: V1/V2/V3 screenshot images manually placed on slides. Shown with audience segmentation data.

Solution: Build an asset library (S3/GCS) where creative team uploads variants tagged by experiment ID. Pipeline pulls matching images automatically.

Strategic Commentary

Current: consultant writes 3-5 bullet points of learnings and next steps. Requires domain expertise.

Solution: AI drafts initial commentary from data trends. Consultant reviews and edits via a simple web form before deck generation.

Campaign Planning Tables

Current: manually maintained in slides with ASA/GAC campaign concepts, markets, and status.

Solution: If structured in Asana/Notion, pull via API. Otherwise, maintain as a structured spreadsheet/BQ table that feeds into the deck.

Point-in-Time Snapshots

Freezing Data for Each Period

Every bi-weekly deck represents a frozen moment. We need to ensure the data used is reproducible and auditable.

Snapshot Strategy

  • - BigQuery scheduled queries snapshot metrics at end of each bi-weekly period
  • - Snapshots stored in snapshots.aso_biweekly_YYYYWW tables
  • - Each deck generation logs which snapshot was used
  • - Can regenerate any past deck from its snapshot

Versioned Outputs

  • - Naming: DC_BiWeekly_FY26Q2_W13-14.pptx
  • - Stored in GCS with version history
  • - Diff view: what changed from last period's deck
  • - Audit trail: who added manual commentary, when

Step-by-Step Build Guide

How to Build This (Using Claude Code)

A practical, day-by-day guide for the data analyst building this out. Each step uses Claude Code as a pair-programming partner.

Day 1: Data Layer Discovery

GOAL: Know exactly what data we have and where it lives

  • 1.1 Clone Merlin's aso-analytics repo - this already has BigQuery connections and schema catalog
  • 1.2 Use Claude Code to explore BigQuery tables - ask it to list all tables, describe schemas, identify which ones contain the data points from the table above
  • 1.3 Map data availability - for each data point above, confirm: is it in BigQuery? What table? What field? What date range? Any gaps?
  • 1.4 Identify legacy tables - flag anything unused or outdated (e.g., organic vs paid split tables from 2019)
  • 1.5 Document findings - Claude Code can help generate a schema documentation MD file as you go

Day 2: Context Layer Setup

GOAL: Teach the AI system Adobe's business logic

  • 2.1 Create context MD files in the repo - these are the knowledge base that Claude Code and any future agents will use:
    context/
      fiscal-calendar.md   # FY start Dec, fiscal week mapping
      market-tiers.md     # P0/P1/P2 countries
      products.md         # Scan, Reader, DC configs
      metrics.md          # FTD, CVR, SP definitions
      glossary.md         # IAE, PCE, CPP, KWO, etc.
      targets.md          # Q2 FY26 quarterly targets
  • 2.2 Build fiscal week calculation - Python function that maps any date to Adobe fiscal week/quarter. Use Claude Code to write and test this.
  • 2.3 Write SQL query templates - parameterized queries for each data point (e.g., "get FTDs for product X, fiscal weeks Y-Z, broken by platform")
  • 2.4 Test queries against BigQuery - validate that the data matches what's currently in the decks

Day 3: First Slides - Performance Recap

GOAL: Auto-generate the highest-value slides that take the most time manually

  • 3.1 Install python-pptx and create a branded slide template matching the Phiture deck format
  • 3.2 Build the FTD bar chart generator - matplotlib chart that matches the existing format: red bars (CY), gray bars (LY), black target line. Use Claude Code to iterate on the styling until it matches.
  • 3.3 Build the YTD comparison table - iOS / Android / Total rows, with vs Target % and vs YoY % columns, color-coded (green up, red down)
  • 3.4 Wire it together - Pull data from BigQuery → apply fiscal week context → generate chart → inject into PPTX template
  • 3.5 Compare output to real deck - open the generated slide next to the manual one. Iterate until they match.

Day 4: Keywords + More Slides

GOAL: Add keyword ranking slides and additional performance breakdowns

  • 4.1 Build keyword ranking table slide - pull VIP and starred keyword data from AppTweak tables in BigQuery. Show rank, delta, search popularity per market.
  • 4.2 Build search/browse breakdown slides - impressions, FTDs, CVR for Search vs Browse vs Explore (iOS) and Search vs Explore (Android)
  • 4.3 Add static config slides - title, managed locales (P0/P1/P2), glossary. These just pull from context MD files.
  • 4.4 Stretch: IAE/PCE event results - if time allows, generate event performance tables by country

Day 5: AI Narrative + Polish + Demo

GOAL: Add AI-generated analysis and prepare a demo

  • 5.1 Wire Claude into the pipeline - after data is pulled, send it to Claude with the context layer to generate: (a) executive summary, (b) key observations, (c) notable trends or anomalies
  • 5.2 Build the review workflow - AI draft is saved as editable text. Consultant reviews, edits, then triggers final deck generation.
  • 5.3 Create the CLI interface: python generate_deck.py --product reader --period 202613-202614
  • 5.4 Run an end-to-end test - generate a full Reader Bi-Weekly deck and compare side-by-side with the real one
  • 5.5 Prepare demo - record a before/after: manual process vs automated. Show the time savings.

Conversational Analysis

JJ
Why did iOS Reader downloads drop last week?
AI
Reader iOS FTDs dropped 8.8% WoW (week 202612). Three likely contributing factors:

1. Paused EMEA paid media - ASA campaigns in DE, FR, IT were paused for budget reallocation, removing ~12K weekly installs
2. Web referral decrease - 15% drop in web referrals, possibly linked to Adobe.com landing page changes
3. Keyword visibility loss - "pdf editor" dropped from #2 to #5 in US (est. -3K weekly installs), "pdf convertor" lost from #3 to #8 in DE

Recommendation: Resume EMEA campaigns ASAP and investigate the web referral drop with the Adobe web team.

⚠️ Anomaly Detected: Scan Android FTDs
Scan Android first-time downloads dropped 12.3% WoW in week 202613, significantly outside the normal ±5% range.
  • Search installs: -18% (primary driver)
  • Explore installs: -3% (within normal range)

  • Action: Check if competitors (CamScanner, Microsoft Lens) show similar drops via Sensor Tower
